# BetaGo-Redefine 交互体验 Review 与优化方向

> 时间：2026-06-23
> 范围：仓库当前主分支 `master`
> 视角：以飞书用户实际感知到的交互链路为主线，从前到后梳理；引用点用 `path:line` 标注，便于直接跳转。

---

## 0. 总览

机器人功能性已经比较完整：消息处理、意图识别、命令系统、卡片回调、todo/schedule、频控、权限、webui 全都有。
但作为一个「在飞书里 daily 用」的产品，仍然存在以下三类共性问题：

1. **反馈通道单薄**：很多链路成功/失败/跳过时用户无感，问题很难自查。
2. **流式 / 异步状态可见性弱**：用户经常看到机器人"憋了几秒再吐一段"，体感像卡死。
3. **危险/高频操作缺二次确认与撤销**，新人 onboarding 也基本没有。

下面按链路展开。

---

## 1. 单条消息回复链路：响应感弱、反馈通道少

### 1.1 静默 skip：用户感知不到机器人「思考过」
- 标准聊天模式下 `decision="skip"` 时直接 `return nil`，没有任何反馈。
  - `internal/application/lark/handlers/chat_handler.go:536-538`
- 随机概率回复未命中时只在日志里打印。
  - `internal/application/lark/messages/ops/chat_op.go:116-128`
- 意图识别已经产出 `interrupt_risk`、`user_willingness` 这些信号，但没用来驱动一个轻量「机器人在听 👂」的低打扰反馈。

**优化方向**：
- 被 `@`/p2p 时，至少给一个 `OnIt → 👀/💭` 的 reaction 表示"看到了，但不发言"，区分"机器人没收到"和"机器人主动选择不接话"。
- 已有的 `withProgressReaction` / `addDoneReactionIfNeeded`（`internal/application/lark/messages/ops/common.go:121-130`）只用在 `ReplyChatOperator`，没覆盖 random / intent 路径，可统一接入。

### 1.2 流式只对 question 类卡片开启，其它都"憋大招"
- `shouldUseStreamingCard` 仅在 `intent=question && NeedReply` 时才走流式卡。
  - `internal/application/lark/handlers/chat_handler.go:275-287`
- 其它路径都是先把整段 JSON 流读完，再 `ReplyMsgText` 一次性发出。
  - `internal/application/lark/handlers/chat_handler.go:518-548`
- 模型在标准对话里要先输出 `thought / reference_from_web / reference_from_history / reply` 整段 JSON 再被解析，普通用户感知到几秒空窗 + 一句话。

**优化方向**：
- 把"流式卡片"做成默认路径（至少 @bot/p2p 时默认走流式卡）。
- 或者让标准 prompt 在长回复里产出可分段的 reply，前端按段 patch 文本消息。

### 1.3 错误链路全部吞掉
- `runStandardChat` 里 LLM/工具调用任何环节失败都是 `return err`；上层 `Run` 里只 log，不给用户回执。
  - `internal/application/lark/messages/ops/chat_op.go:108-128`
  - `internal/application/lark/messages/ops/reply_chat_op.go:69-86`
- 历史检索 / `RecallDocs` 失败只 `Warn` 后继续，用户看到的就是机器人"罢工"。
  - `internal/application/lark/handlers/chat_handler.go:617-619`

**优化方向**：
- 建立统一的"轻量错误回执 = reaction `❗` + 简短文本气泡 + 可重试按钮"模式，分别落到 reply_chat、command、card_action 三条链路。

### 1.4 纠错只能靠自然语言、成本高
- 纠错走 `store_correction` 工具调用，每次都要再触发一次 LLM tool roundtrip。
  - `internal/application/lark/handlers/chat_handler.go:421-422`
  - `buildCorrectionsContext` 把纠错注入到下次 system prompt
- 用户没有"👍/👎/请改"的低成本反馈通道，每次纠错都要打字。

**优化方向**：
- 在 chat 回复气泡上挂一行可选的反馈按钮（👍/👎/这里错了→打开纠正表单卡）。
- 把"纠正"从"再说一段话"降成"一次点击 + 可选输入"，并直接复用 `chat_correction_handler` 的存储路径。

## 2. 命令系统：可发现性还可以，但深用就割裂

### 2.1 命令树长、根 help 卡缺搜索
- 根 help 一次性把所有子命令塞成按钮组。
  - `internal/application/lark/command/help.go:203-234`
- 命令一多面板要滚很久；没有"按描述模糊搜索"，例如用户搜 `/help 提醒` 不会跳到 `schedule`。

**优化方向**：
- 根 help 加输入框 + 客户端过滤 / callback 用 LLM 匹配。
- 对命令支持别名同义词（`提醒→schedule`、`闭嘴→mute`、`点歌→music`）。

### 2.2 Form 卡和命令是双轨的
- `command.open_form` 走卡片填参数，`command.submit_form` 再回到普通命令链路。
  - `internal/application/lark/cardaction/builtin.go:225-261`
- 机制本身已经比较干净，但用户实际进入门槛仍然是"先要知道 `/help`"。

**优化方向**：
- 在 @ 机器人 + 自然语言里识别"我想点歌 / 我想加个提醒"，由意图识别直接把对应命令的参数表单卡发给用户，跳过 `/help` → `/foo --help` 的两跳。
- 意图识别已经能区分 `IntentTypeCommand`（`internal/application/lark/intent/recognizer.go:71-75`），目前只是没拿来路由 form。

### 2.3 危险操作没有二次确认
- `/config delete`、`/schedule delete`、`/feature block`、`/permission revoke` 都是一次点击直接生效。
  - `internal/application/lark/cardaction/builtin.go:50-59`
- 有 `ActionHistory` 记录，但 UI 上没有"撤销最近一次操作"的入口。

**优化方向**：
- 删除/撤销类按钮统一加 confirm 弹窗（飞书原生 `confirm` 字段就有）。
- 卡片 footer 加"撤销最近一次变更"按钮，复用已经写好的 `PendingHistory` 列表来反向 apply。

## 3. 卡片体验：交互完整，但缺"状态可见性"

### 3.1 长列表分页 / 异步刷新现在都靠用户重点击
- 音乐列表卡走 `card_json` 流式 patch，机制不错。
  - `internal/application/lark/cardaction/builtin.go:145-211`
- 但分页按钮没有 loading state；用户连点会触发 `CancelMusicListStream`（同上文件 `175`），但前端看不到取消反馈。

**优化方向**：
- 分页 / 翻页按钮统一加 disable + skeleton。
- 把"已加载第 N 页 / 共 M 条"放到 footer，避免用户怀疑卡死。

### 3.2 Schedule 编辑用 token + 二次卡片
- `pending_edit` 用 token 暂存，`edit_token` 过期才提示。
  - `internal/application/lark/cardaction/builtin.go:481-557`
- token 失效现在直接 toast `编辑令牌已过期，请重新发起编辑`，没有"重新发起编辑"按钮，用户得回去找原命令重新触发。

**优化方向**：
- toast 里附带一个"重新发起编辑"action，直接以同 ID 重建编辑卡。

### 3.3 卡片 Action History 单向膨胀
- `PendingHistory` 一直 append。
  - `internal/application/lark/schedule/card_view.go:34-42` 等多处
- 没看到截断 / 折叠策略。

**优化方向**：
- 默认只展示最近 3 条 + 折叠面板，避免老群操作多了卡片底部全是历史。

## 4. Mute / 频控 / 反复读：观察性弱

### 4.1 Mute 状态不可见
- `Mute Handler` 只在执行的瞬间回一行文本，「现在被禁言到几点几分」需要靠用户记。
  - `internal/application/lark/handlers/mute_handler.go:73-118`
- `ChatMsgOperator` / `RepeatMsgOperator` 通过 redis key 存在与否判定，用户没法"查一下还有多久解除"。
  - `internal/application/lark/handlers/chat_handler.go:475-487`
  - `internal/application/lark/messages/ops/repeat_op.go:67-72`

**优化方向**：
- mute 设置后回一张轻卡，含"剩余时间 / 立即取消 / +5min / +30min"按钮。
- `/mute status` 子命令对应同一张卡。

### 4.2 频控决策对用户黑盒
- 决策结果只对 admin 通过 `/ratelimit stats` 可见。
  - `internal/application/lark/ratelimit/README.md:152-167`
- 普通用户看到的现象就是"机器人这段时间不爱搭理我"。

**优化方向**：
- 被频控拒绝后给一个低打扰 reaction（如 🥱/🤐）。
- 允许 group admin 在 stats 卡上一键"立即放开冷却"。`StatsCardOptions` 已有 `MessageID + PendingHistory`，扩展一个 `cooldown.reset` action 就能落。

### 4.3 RepeatMsgOperator 对自身不防呆
- `repeat_op.Run` 只看概率和定制配置。
  - `internal/application/lark/messages/ops/repeat_op.go:85-156`
- 没排除"机器人自己发的消息"或"刚刚已经被 repeat 过的消息"，高活跃群里容易反复复读。

**优化方向**：
- 加一层 dedup（最近 N 秒内同 chat 同内容只 repeat 一次）。
- 来源校验：不 repeat sender = bot self。

### 4.4 关键词回复匹配 panic 直接抛上来
- `CheckQuoteKeywordMatch` 命中 default 直接 `panic("unknown match type"...)`。
  - `internal/application/lark/messages/ops/word_reply_op.go:117-128`
- 上层只能靠全局 panic recover（`internal/application/lark/messages/handler.go:64-67`）才不至于挂掉服务。

**优化方向**：
- 换成 `return false + log`；错配置不应该让用户感知到机器人崩溃。

## 5. 个性化 / 群体验：可玩性还没释放

### 5.1 群专属人设有，但用户级偏好缺失
- 持久化的只有 `KeyChatPersona` / `KeyHistoryCutoffTime` / `KeyChatExtraContext`，都是 chat 维度。
  - `internal/application/lark/handlers/chat_handler.go:73-116`
- 同一个群里 A 想要严肃模式、B 想要皮一点 —— 当前做不到。

**优化方向**：
- 扩展 `appconfig` 的 scope 到 user，用 `/bb prefer --tone=serious`、`/bb prefer --reasoning=auto` 之类落到 `permission_grants` 同级表。
- prompt 拼接侧已经有 `getChatPersona` 注入点，加一段"用户偏好"即可。

### 5.2 `/bb forget` 入口太隐蔽
- 设置历史 cutoff 是 `/bb forget`。
  - `internal/application/lark/command/command.go:139-142`
- 普通用户基本不会发现，而上下文重置在 IM 里是高频需求。

**优化方向**：
- 在每次 chat 回复卡片底部放一个"🧹 清空对话上下文"轻按钮。
- 同步把 `KeyHistoryCutoffTime` 写入。

### 5.3 自然语言里没"轻命令"
- `IntentTypeCommand` 已经识别出来。
  - `internal/application/lark/intent/recognizer.go:71-75`
- 但目前没用来主动跳转命令；用户得真的打 `/xxx`。

**优化方向**：
- 意图判定为 command 时，回一张"你是不是想 /music 帮你搜歌？[去填表] [取消]"的卡片。
- 比强行 LLM 兜底体验顺滑很多，也省 token。

## 6. 首次接入 / Onboarding

- `bootstrap_admin_open_id` 必须在 `config.toml` 里手填，第一批权限不会自动写库。
  - `README.md:401-408`
- 没有 `welcome` 卡片：被拉进群、第一次 P2P 都没提示"我能干什么、@我试试"。
- `script/migrations` 只覆盖 todo/schedule 演进；新表（`prompt_template_args` 必填两条种子）只在 README 里说明。
  - `README.md:284-289`

**优化方向**：
1. 第一次接收 `im.message.receive_v1` 时，识别"机器人首次进入这个 chat"，自动发一张 onboarding 卡（介绍 `/help` `/bb` `/music` `/schedule`）。
2. 启动阶段对必填种子数据做断言并打印明确错误，避免后台报错堆里找原因。
3. bootstrap admin 第一次进入 `/permission` 时直接落库一条 `permission.manage@global`，并 toast「已写入，可关闭 bootstrap_admin_open_id」。

## 8. 优化方向（按 ROI 排）

1. **回复反馈通道**：被 mention/p2p 但 skip → 给一个 reaction；错误 → 一句兜底气泡 + reaction；chat 回复挂 👍/👎/纠正按钮。
2. **流式默认化**：把卡片 + 流式从 question-only 推广到所有需要回复的链路；至少 @bot 路径默认走流式卡。
3. **自然语言 → 命令路由**：意图为 command 时直接弹对应命令的 form 卡；同步引入命令同义词 / 模糊搜索。
4. **危险操作二次确认 + 撤销**：feature/config/schedule/permission 删改加 native confirm，footer 加"撤销最近一次"。
5. **Mute / 频控可见化**：mute 状态卡 + 频控被拒时低打扰 reaction，admin 卡上加 reset cooldown。
6. **Onboarding & Bootstrap**：首次进群 / p2p 自动发欢迎卡；bootstrap admin 自动落库；启动期断言种子数据。
7. **用户级偏好**：把 persona / tone / reasoning 偏好下放到 user scope，避免一群多人风格冲突。
8. **卡片防呆**：分页 loading / Action history 折叠 / pending_edit 过期可一键重发。
9. **小修缮**：去掉 `SetSkipDone(true)` 一律覆盖、去掉 panic、替换被注释掉的 sleep、help not-found 加模糊匹配。
10. **i18n & 文案抽离**：emoji + 中文硬编码集中到一个 copywriting 包，便于后续多语言 / 换皮肤。

---

## 9. 后续动作建议

- 按优先级 1~3 在下一次迭代里挑 1~2 条落地，先把"反馈通道 + 流式默认化"这两件事铺开，对体感提升最大。
- 优先级 4~6 涉及交互改造，建议在改造前画一份卡片态机草图（confirm / 撤销 / mute 状态卡），避免一次落多张卡导致风格分裂。
- 优先级 7~10 偏内功，可以拆成独立小 PR 持续清理。
